# Okron Web — i18n Rules

## Non-negotiable rule

**Every string visible to the user must go through `useTranslations`.** No exceptions — not for labels, buttons, placeholders, error messages, tooltips, or empty-state text.

```typescript
// Correct
const { t } = useTranslations();
<p>{t('update.customer')}</p>
<label>{t('date')}</label>
<button>{t('create.workorder')}</button>

// Wrong — hardcoded strings in JSX
<p>Actualitzar Client</p>
<label>Data</label>
<button>Crear OT</button>
```

## Key convention

Keys use `dot.notation` in **lowercase**. Two or three segments are the norm.

| Pattern | Example key |
|---|---|
| CRUD action + entity | `create.workorder`, `update.customer`, `delete.sparePart` |
| Single concept | `date`, `status`, `description` |
| State or label | `state.pending`, `state.completed` |
| Section heading | `section.assets`, `section.preventive` |

Do not use camelCase, PascalCase, or underscores in keys. `create.workOrder` is wrong — use `create.workorder`.

## Where translations are stored

Translations live in **MongoDB**, managed via the admin panel. There are no `.json` translation files in the repo. Do not create them.

## Adding new keys — workflow

When writing code that introduces a new `t('some.key')` call:

1. Use the key in the code — `t(key)` returns the key itself if no translation exists, so the app does not break.
2. **Report every new key to the user explicitly** with its Catalan value. Use this format:

```
New i18n keys to register in MongoDB (Catalan):
- `create.downtime` → "Crear parada"
- `downtime.reason` → "Motiu de la parada"
- `downtime.duration` → "Durada"
```

3. Do not mark a feature as complete until the user has confirmed the keys are registered.

## What happens when a key is missing

`t(key)` returns the key string itself (e.g., `"create.downtime"`). The UI renders the raw key — ugly but not broken. This is intentional and by design. Do not add fallback logic or conditional rendering around `t()` calls.

## Hook usage

`useTranslations` is a custom hook. Use it at the top of the component or hook, not inside loops or conditions.

```typescript
// Correct — called at component top level
function DowntimeForm() {
  const { t } = useTranslations();

  return <label>{t('downtime.reason')}</label>;
}

// Wrong — called conditionally or inside a loop
function DowntimeList({ items }: Props) {
  return items.map(item => {
    const { t } = useTranslations(); // violates rules-of-hooks
    return <span>{t('downtime.duration')}</span>;
  });
}
```

## Dynamic values in translations

If the translated string includes a dynamic value (e.g., a count or name), the current pattern is string interpolation — not i18n interpolation placeholders:

```typescript
// Current pattern (acceptable)
`${t('asset.count')}: ${assets.length}`

// Do not introduce ICU message format or {variable} placeholders without explicit approval.
```
